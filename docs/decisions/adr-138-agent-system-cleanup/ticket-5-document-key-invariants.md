# Document task and relay key invariants

`task-persistence.ts` has a non-obvious dual-key model: tasks are indexed by `agentSessionId` (the Map key) but referenced externally by `id` (the UUID). After ticket 1, there's also `idIndex`. The relay (`hook-relay.ts`) has its own pair: `paneRootSessionMap` and `sessionStateMap`, with a hard namespace boundary (agent UUIDs vs pane IDs) that ADR-133 just fixed in one place but exists everywhere.

See ADR-138 §"Change 5" for context.

## What to change

Add concise comment blocks at the top of two files. Goal: a future contributor reading either file should understand the key model in 30 seconds.

### `electron/task-persistence.ts` — add at the top of the file (after imports)

```ts
/**
 * Task persistence model
 *
 * A TaskInfo has three identifiers; they exist for different consumers:
 *
 * - `id`        — UUID, the public stable handle. Used by IPC, by the renderer
 *                 store, and by any code that needs to refer to a task across
 *                 status changes. Generated by `crypto.randomUUID()`. Never
 *                 mutated after creation.
 * - `agentSessionId` — the agent CLI's session UUID (Claude's `session_id`,
 *                 Codex's session id, etc.). Set once at creation from the hook
 *                 payload. Used as the *internal Map key* because the hook relay
 *                 only knows the agent session id. Never mutated.
 * - `paneId`    — the daemon's pane-keyed session id (== paneId on the renderer).
 *                 Nullable. Cleared when the task's pane is closed or when the
 *                 relay reassigns the pane to a new agent session.
 *
 * Note: `agentSessionId` and `paneId` live in *different namespaces* and are
 * never interchangeable. Mixing them up is the bug fixed by ADR-133.
 *
 * Lookup tables:
 * - this.tasks    — Map<agentSessionId, TaskInfo>  (primary store, persisted)
 * - this.idIndex  — Map<id, agentSessionId>        (in-memory only, rebuilt from tasks)
 *
 * On every mutation, both maps must agree. Ticket-1 invariant tests cover this.
 */
```

### `electron/hook-relay.ts` — add at the top of the file (after the existing module doc comment)

```ts
/**
 * Identifier namespaces inside the relay
 *
 * - `paneId`     — the daemon's session id (== paneId on the renderer side).
 *                 Lifetime: created on `pty:create`, destroyed on `pty:close`.
 * - `sessionId`  — the agent CLI's session UUID, extracted from hook payloads.
 *                 Lifetime: created on `SessionStart`, destroyed on `SessionEnd`.
 *                 One pane can host multiple sessions over its lifetime
 *                 (`/clear`, `claude --resume`).
 *
 * Two maps:
 * - paneRootSessionMap: Map<paneId, sessionId>
 *     The current root session on each pane. Used to ignore subagent
 *     SessionStart events and to resolve the pane's *current* task when a hook
 *     event arrives without enough context.
 * - sessionStateMap: Map<sessionId, SessionState>
 *     Per-agent-session bookkeeping (active subagents, pendingStopAt, etc).
 *     Cleared on SessionEnd, on SessionStart replacement, and by stale-orphan
 *     sweeps.
 *
 * Always carry the namespace explicitly: the bug fixed by ADR-133 was a sweep
 * comparing a sessionId against a paneId set.
 */
```
